# hex

A drop-in fast path for lowercase hexadecimal (base16) encoding, byte-identical
to the standard library's [`encoding/hex`](https://pkg.go.dev/encoding/hex). The
bulk of **encoding** runs a SIMD kernel generated by
[go-asmgen](https://github.com/go-asmgen/asmgen); the short tail reuses the
standard library, so output matches exactly. Pure Go, `CGO_ENABLED=0`, stable Go,
no `GOEXPERIMENT`.

```go
s := hex.EncodeToString(data)         // same bytes as encoding/hex.EncodeToString
b, err := hex.DecodeString(s)         // same bytes + same errors as encoding/hex
```

API mirrors `encoding/hex`: `Encode`, `EncodeToString`, `Decode`,
`DecodeString`, `EncodedLen`, `DecodedLen`.

| op | amd64 | arm64 / loong64 / riscv64 |
|---|---|---|
| **encode** | **SSE2/SSSE3** + **AVX2** (runtime dispatch) | scalar (stdlib) — NEON/LSX/RVV planned |
| **decode** | scalar (stdlib) — SIMD planned | scalar (stdlib) |

## How encode works

Per 16 input bytes (SSE; 32 for AVX2): split each byte into its high and low
nibble (`PSRLW $4` + `PAND 0x0f` for the high nibble, `PAND 0x0f` for the low),
map each nibble `0..15` to its ASCII digit via a `PSHUFB` lookup of the table
`"0123456789abcdef"`, then interleave the two nibble streams with
`PUNPCKLBW`/`PUNPCKHBW` so each input byte's `(hi, lo)` characters land adjacent,
and store. The AVX2 path doubles the width and recombines the two 128-bit lanes
with `VPERM2I128`. Constants are emitted via go-asmgen's `emit.File.Data`.
Verified against `encoding/hex` (table + fuzz, native arm64 + amd64 under
Rosetta).

**Decode** currently delegates to `encoding/hex` (scalar), so `Decode` /
`DecodeString` are byte- **and** error-identical to the standard library —
including `InvalidByteError` at the exact offset and `ErrLength` on odd input. A
SIMD decode kernel (the fiddly part is byte-exact validation/error offsets) is
planned. Correctness first.

## Performance

Encode/decode throughput on a 1 MiB random buffer, **native amd64** (GitHub
Actions `ubuntu-latest`), `-count=6`, median MB/s — see
[`.github/workflows/bench.yml`](.github/workflows/bench.yml). The dev box is
arm64, where this package's amd64 kernel only runs under Rosetta
(unrepresentative), so these numbers come from CI on native hardware (GitHub
Actions `ubuntu-latest`, AMD EPYC, AVX2-capable, `GOAMD64=v1`), median of 6.

| implementation | kind | encode MB/s | vs stdlib | decode MB/s | vs stdlib |
|---|---|---:|---:|---:|---:|
| `encoding/hex` (stdlib) | scalar | 929 | 1.00× | 1885 | 1.00× |
| **this package** | pure-Go SIMD (SSE2/SSSE3 + **AVX2**) encode; stdlib decode | **21961** | **23.6×** | 1885 | 1.00× |
| [`tmthrgd/go-hex`](https://github.com/tmthrgd/go-hex) | pure-Go SIMD (SSE/AVX), **archived** | 19982 | 21.5× | 8644 | 4.59× |

This package's forced SSE-only encode path is ~15700 MB/s (16.9×); the AVX2
dispatch above is the default on AVX2 hardware.

Honest notes:

- [`tmthrgd/go-hex`](https://github.com/tmthrgd/go-hex) is the prior pure-Go SIMD
  hex codec (hand-written Plan9 asm, SSE/AVX). It was **archived upstream in
  September 2025**. It also ships a **SIMD decoder**, which this package does not
  yet — so on decode it is meaningfully faster than our stdlib-delegating path.
  On encode the two are comparable. It is the obvious decode target to match.
- Go's standard `encoding/hex` is scalar (byte-by-byte); see
  [golang/go#68188](https://github.com/golang/go/issues/68188), the open proposal
  to add SIMD to the standard library.

## Regenerating the assembly

`encode_amd64.s` is committed. To regenerate (go-asmgen is a build-time tool, not
a runtime dependency):

```sh
go get github.com/go-asmgen/asmgen@v0.4.0
go run encode_gen.go
go mod edit -droprequire github.com/go-asmgen/asmgen
go mod tidy
```
